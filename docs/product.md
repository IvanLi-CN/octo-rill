# OctoRill 内部产品参考

这份文档供 specs、实现讨论和协作者同步产品口径时使用。公开概览请看 `docs-site/docs/product.md`；这里保留更完整的行为语义和实现边界。

## 产品定位

OctoRill 面向需要持续查看 GitHub 动态的个人用户。它把“与我直接相关”的信息收束成一个阅读工作区：关注仓库的发布更新、他人对我的社交反馈、按固定窗口生成的日报，以及 GitHub Inbox 入口。

目标不是替代 GitHub，而是减少以下摩擦：

- 为了追发布更新频繁切换仓库页、Release 页和通知页
- 想快速理解 release 内容时只能直接啃原文
- 需要回顾最近一轮变化时，没有一个稳定的个人视角窗口

## 主要对象与信号

### Release

Release 是主阅读对象，承担最多的衍生能力：

- **全部（聚合）**：按日报边界分组展示 Dashboard 活动流；当前日组直接显示原始活动卡片，历史日组默认优先显示对应日报，并允许在“日报视图”和“日期分界 + 原始活动列表”之间单独切换。历史日报会紧凑概括同窗内的 Release 及实际发生的获星 / 关注；在日报视图下，已被摘要覆盖的社交卡片不再重复出现，切回列表视图后仍可查看同日原始混排活动卡片。
- **发布**：左侧展示按日报边界分组的 Release 信息流（Feed）；仅在日组切换处显示弱化日期分隔线，首个可见日组不显示前置分隔，右侧展示「Inbox 快捷入口」。
- **加星**：左侧仅展示“谁给我的个人仓库加星”的社交活动卡片，按同一日边界分组；右侧展示「Inbox 快捷入口」。
- **关注**：左侧仅展示“谁关注了我”的社交活动卡片，按同一日边界分组；右侧展示「Inbox 快捷入口」。
- **日报**：左侧展示日报内容查看（并支持手动生成）；右侧展示「日报列表 + Inbox 快捷入口」。
- **收件箱**：左侧展示通知列表（用于查看与跳转），不进入 Feed；右侧仍展示 Inbox 快捷入口。

### 社交动态

当前只覆盖两类“与我直接相关”的社交反馈：

- `repo_star_received`
- `follower_received`

这两类卡片保持轻量，只展示必要身份信息、时间和 GitHub 跳转入口，不引入 AI lane。

### Daily brief

日报是按固定时间边界生成的回顾视图，不是长期归档报告。它把一个窗口内的变化重组为两部分：

- 时间窗口：以 `AI_DAILY_AT_LOCAL` 设定的本地时间（例如 `08:00`）为边界，
  窗口为「昨日 08:00（本地）→ 今日 08:00（本地）」。
- 默认边界：若未设置 `AI_DAILY_AT_LOCAL`，默认按本地 `08:00` 计算窗口。
- 内容结构固定为：
  - `## 项目更新`：始终存在，按项目分组逐条列出 release 与变更要点；若窗口内没有 Release，则显示 `- 本时间窗口内没有新的 Release。`
  - `## 获星与关注`：仅当同一窗口内实际存在 `repo_star_received` 或 `follower_received` 动态时出现；只渲染有数据的 `### 获星` / `### 关注` 小节，不再输出空社交占位
- 链接策略：
  - release 主链接：站内链接 `/?tab=briefs&release=<release_id>`，点击后在 briefs 页展示详情卡
  - 其他链接：跳转 GitHub（仓库页、release 原文页、相关 GitHub 链接）；相关链接显示文本应尽量短，优先使用 `#123`、短 SHA 等 GitHub-aware 标签，避免正文过长
- 正文约束：
  - 卡头负责展示日期、时间窗口与边界信息；正文不再重复输出时间窗口/项目计数等概览元数据
  - 落库前会剥离整篇外层 Markdown code fence，避免整篇日报被当作代码块渲染
- 链接完整性保障：日报润色后会按 `release` 查询参数做精确 `release_id` 校验，缺失时自动补链（避免 `12/123` 前缀误判）。
- 用途：让用户快速了解这段时间发生了什么，并能一跳查看完整上下文。

## 前台界面结构

### Landing

Landing 负责公开入口和登录承接：

- 解释产品在整理什么信息
- 说明它不是什么
- 提供 GitHub OAuth 登录入口

不在 Landing 里展开复杂操作说明或配置细节。

### Dashboard

Dashboard 采用主列 + 侧栏结构，核心标签为：

- `全部`
- `发布`
- `加星`
- `关注`
- `日报`
- `收件箱`

右侧侧栏长期承担 Inbox 快捷入口；在 `日报` 标签下还同时承担日报列表。

#### 全部

用统一时间线混排发布更新、获星与关注动态。历史日组会优先折叠成日报卡片；当用户切换到列表视图时，再展开原始活动卡片。

#### 发布

只展示 Release 信息流，并按日报边界分组。用户在这里主要做“连续阅读”和“切换 lane”。

#### 加星 / 关注

只保留社交动态条目，目标是让用户快速判断“谁在给我反馈”，而不是扩成完整社交 feed。

#### 日报

左侧是日报阅读与手动生成，右侧是日报列表和 Inbox 快捷入口。

#### 收件箱

站内只做通知入口与列表，不接管 GitHub 完整通知工作流。

## Release 阅读模型

发布条目支持 `原文 / 翻译 / 智能` 三种阅读模式：

- 默认优先帮助用户快速扫读
- 保留原文作为最终核对入口
- 在日报中，Release 主入口进入站内详情弹窗，而不是把正文重新塞回主列

详情弹窗保留 Markdown 结构，并提供 GitHub 外链按钮查看原始 Release。

## 翻译策略与需求窗口

AI 翻译不是一次性批处理后静态展示，而是结合同步预热与前端可见窗口触发：

- 同步 release 时，为新增内容补后台翻译任务
- 前端根据真实可见窗口和后续连续卡片，批量请求结果状态
- 已有 ready 译文时，后续 refresh 不会先把界面打回空白“翻译中”
- 自动轮询不会无脑重试 `error`；显式用户动作才触发重试

目标是让用户在真实阅读路径上尽快得到可展示结果，而不是把翻译系统设计成另一个需要单独操作的工作台。

## 日报窗口与时区

- 默认日报边界：`08:00`
- 计算使用服务端约定的本地时间与时区，而不是浏览器瞬时时区
- 支持管理员与运行时配置影响默认时区与调度窗口

这个约束的意义是保证“昨日更新”在不同设备上仍有稳定口径。

## 权限模型

### GitHub OAuth

OAuth 负责：

- 登录
- Feed / Notifications / Starred / Releases / Followers / owned repo stargazers 同步

OAuth 默认走最小权限，不为站内 reaction 写操作额外申请权限。

### GitHub PAT

PAT 只用于 Release feedback 写操作，和 OAuth 通道分离：

- 未配置 PAT：reaction 只读
- 已配置且有权限：允许站内点按 reaction

当用户点击 reaction 且未配置 PAT 时，必须通过对话框引导配置，而不是只在卡片底部报错。

## 数据语义

- SQLite 是本地缓存与派生层，不是最终事实源
- Release 以稳定 `release_id` 为共享事实主键
- Star 影响的是“当前用户看哪些列表/日报候选”，不决定 release 详情本身是否仍存在

因此：

- 列表和日报具有个性化可见范围
- 详情读取与详情翻译遵循共享事实语义

## 后台与运维面

管理员任务中心主要处理三类问题：

- 同步与调度是否在正常推进
- 翻译批次、请求与 worker 是否健康
- LLM 调用、并发与输入预算是否符合当前运行配置

这些页面服务于可观测性与运行时控制，不属于普通用户的阅读主路径。

## 非目标

- 不做完整 GitHub 客户端
- 不在站内承接评论、合并、通知已读等完整交互
- 不扩成多语言产品设置页；当前文案口径固定中文
- 不依赖持续后台全量自动同步来塑造主体验，用户仍可主动触发同步
