# OctoRill 产品说明

OctoRill 是一个面向个人 GitHub 动态的聚合与阅读界面：提供一个 **Dashboard 活动流**，集中展示 **发布更新（Releases）**、**谁给我的个人仓库加星**、**谁关注了我** 三类与“我”直接相关的动态；同时提供 **发布内容中文翻译与要点整理**、**Release 日报** 与 **Inbox（GitHub Notifications）** 的快捷入口。日报中的 release 主入口会在站内打开详情弹窗，其余外链统一跳转 GitHub。

## 核心体验

- **信息流（Feed）**：在 `全部` tab 中按时间倒序混排发布更新、仓库被加星、账号被关注三类动态，并支持无限滚动加载更多。
- **发布阅读模式**：Release 条目支持 `原文 / 翻译 / 智能` 三种阅读模式；发布内容可查看中文译文与要点整理，必要时再切回原文。
- **Release 日报**：根据固定时间边界生成“昨日更新”日报，按项目分组覆盖完整 release 内容，并输出可点击链接。
- **Inbox 快捷入口**：把 GitHub Notifications 抄一份到侧栏，提供快速跳转入口。
- **个人社交动态**：支持查看“谁给我的个人仓库加星”“谁关注了我”，并在卡片中展示对方头像、账号与 GitHub 跳转入口。

## 页面结构（Web）

### Tabs：全部 / 发布 / 加星 / 关注 / 日报 / 收件箱

页面整体采用**固定左右两栏**布局：

- **左侧主列**：根据 Tab 展示对应内容（聚合 Feed / 发布信息流 / 社交动态 / 日报 / 收件箱列表）。
- **右侧侧栏**：展示“其他内容”，在看 Feed（全部 / 发布 / 加星 / 关注）时包含：
  - **Inbox 快捷入口**
  在「日报」Tab 下：右侧包含「日报列表 + Inbox 快捷入口」；在「收件箱」Tab 下：右侧保留 **Inbox 快捷入口** 作为常驻入口。

Tab 语义：

- **全部（聚合）**：按日报边界分组展示 Dashboard 活动流；当前日组直接显示原始活动卡片，历史日组默认优先显示对应日报，并允许在“日报视图”和“日期分界 + 原始活动列表”之间单独切换。历史日报会紧凑概括同窗内的 Release、获星与关注；切回列表视图后仍可查看同日原始活动卡片。
- **发布**：左侧展示按日报边界分组的 Release 信息流（Feed）；仅在日组切换处显示弱化日期分隔线，首个可见日组不显示前置分隔，右侧展示「Inbox 快捷入口」。
- **加星**：左侧仅展示“谁给我的个人仓库加星”的社交活动卡片，按同一日边界分组；右侧展示「Inbox 快捷入口」。
- **关注**：左侧仅展示“谁关注了我”的社交活动卡片，按同一日边界分组；右侧展示「Inbox 快捷入口」。
- **日报**：左侧展示日报内容查看（并支持手动生成）；右侧展示「日报列表 + Inbox 快捷入口」。
- **收件箱**：左侧展示通知列表（用于查看与跳转），不进入 Feed；右侧仍展示 Inbox 快捷入口。

### 主列：发布信息流（无限滚动）

- 分组边界：按 Dashboard 当前使用的日报边界时间切分日组，默认 `08:00`；时区以服务端配置为准，不跟随浏览器本地时区漂移。
- 组头样式：仅在日组切换处使用弱化分隔线显示日期与当日 Release 数；首个可见日组不显示前置分隔。历史日报组需要保留卡片形态，但卡头要直接继承这套 divider 的视觉语言，避免“分隔线”和“日报卡片”割裂。
- `全部` tab 的历史组折叠规则：若该日组有对应真实日报，则默认显示日报卡片；点击卡头右侧的「列表」后，当前组切换为“日期分界 + 原始 Release 列表”，并在分界右侧显示「日报」按钮用于切回日报视图。
- `全部` tab 的历史组生成规则：若该日组没有对应日报，则默认显示“日期分界 + 原始 Release 列表”，并在分界右侧显示「生成日报」按钮；点击后按钮进入 spinning 状态，同时用占位日报卡片承接等待过程，待生成成功后替换成真实日报。

信息流条目包含两种：

1) **Release 条目**
- 展示：仓库名、发布版本（tag 或 release name）、发布时间、正文内容（按统一字符上限裁切后的 Markdown）、（可选）中文翻译标题与正文译文 / 智能要点
- 阅读模式：页面支持 `原文`、`翻译`、`智能` 三种 lane；默认页级模式为 `智能`，但用户可以切换为原文或翻译
- 反馈：展示 GitHub 同款反馈表情（👍 😄 ❤️ 🎉 🚀 👀），支持查看计数；站内点按切换需要用户提供 PAT
- 操作：点击“在 GitHub 打开”跳转到对应 Release 页面（新标签页）

2) **社交动态条目**
- 类型：`repo_star_received`、`follower_received`
- 展示：对方头像、login、发生时间；若为 `repo_star_received`，额外展示被加星的个人仓库名
- 反馈：无 AI lane、无 reactions，保持只读轻量卡片
- 操作：点击“GitHub”跳转到对方 GitHub 主页

### 日报（独立 Tab）

- 时间窗口：以 `AI_DAILY_AT_LOCAL` 设定的本地时间（例如 `08:00`）为边界，
  窗口为「昨日 08:00（本地）→ 今日 08:00（本地）」。
- 默认边界：若未设置 `AI_DAILY_AT_LOCAL`，默认按本地 `08:00` 计算窗口。
- 内容结构固定为：
  - `## 项目更新`：按项目分组逐条列出 release 与变更要点
  - `## 获星与关注`：紧凑汇总同一窗口内的 `repo_star_received` 与 `follower_received` 动态
- 链接策略：
  - release 主链接：站内链接 `/?tab=briefs&release=<release_id>`，点击后在 briefs 页展示详情卡
  - 其他链接：跳转 GitHub（仓库页、release 原文页、相关 GitHub 链接）；相关链接显示文本应尽量短，优先使用 `#123`、短 SHA 等 GitHub-aware 标签，避免正文过长
- 正文约束：
  - 卡头负责展示日期、时间窗口与边界信息；正文不再重复输出时间窗口/项目计数等概览元数据
  - 落库前会剥离整篇外层 Markdown code fence，避免整篇日报被当作代码块渲染
- 链接完整性保障：日报润色后会按 `release` 查询参数做精确 `release_id` 校验，缺失时自动补链（避免 `12/123` 前缀误判）。
- 用途：让用户快速了解这段时间发生了什么，并能一跳查看完整上下文。

### Release 详情弹窗（从日报或列表进入）

- 入口：日报中的 release 主链接（站内链接）触发打开；详情以模态弹窗呈现，不再把正文内联插回主列。
- 内容：
  - 默认显示完整中文翻译（支持切换原文）
  - 保留 Markdown 结构（标题、列表、代码块、表格、链接）
  - 提供 “GitHub” 外链按钮查看原始 release 页面

### 右侧侧栏：Inbox 快捷入口

- 展示最近的通知条目（优先未读）。
- 用途：作为快速入口方便用户查看；不试图替代 GitHub 的完整通知工作流。

## AI 翻译策略

- 默认语言：中文（`zh-CN`）。
- 触发方式：
  - **同步后后台预热**：`sync.releases`、`sync.all` 的 release 阶段，以及订阅同步写入新 release 后，会自动为本轮新增 release 追加后台翻译任务。
  - **懒加载自动**：前端会基于真实 DOM 视口几何持续重算 demand window：把当前视口内所有可见 release，以及最后一个可见卡片之后的连续 10 条 release，按原顺序交给结果聚合接口；若单次候选数超过接口的 60 条上限，则前端会继续顺序拆批。
- 启动时机：feed 自动翻译只保留一个很短的聚合窗口，默认约 `500ms`；页面打开后会尽快把当前可见窗口送入队列，而不是长时间停留在“排队中”。
- 结果聚合策略：前端不再直接创建 feed 翻译 request，而是请求“这批 release 当前的译文状态”；后端先读结果表，再决定是否复用 work item 或补建队列需求。
- 正文口径：feed 自动翻译的输入是卡片中展示的正文 Markdown，而不是摘要摘录。
- 超长正文策略：当原始 release 正文超过单次翻译上限时，列表仍显示截断后的原文正文，但自动翻译直接落终态失败，不做分块翻译。
- 终态保留策略：若结果表已经记录了当前 source hash 的 `missing/error` 终态，页面刷新不会自动再次排队；只有新的显式需求（例如手动点“翻译”）才会重新进入 ensure 链路。
- 结果保留策略：如果某条 release 已经有一份可展示的 ready 译文，而更新后的 source hash 又进入排队，界面会继续显示旧译文，直到新任务真正完成；不会先把卡片打回空白“翻译中”。
- 普通读取一致性：无论是 feed 卡片还是 release detail，只要 refresh 还在途，读取接口都继续回旧的 ready 译文；不会因为页面刷新就把现有译文清空。
- 旧页面容错：若旧页面稍后又拿着旧 source 请求译文，后端会先按服务器当前 release 内容重建 canonical source，再决定复用还是排队；旧 source 不会把新译文覆盖回去。
- 批处理策略：后端仍会把多个 release 合并进同一批翻译，但单批输入预算会被稳定限制在约 `1800 tokens`，以便多个 worker 并行消化，而不是把几十条 release 挤进 1-2 个超大批次。
- 同源重复请求：同一用户反复打开页面、轮询或重复触发同一批 release 时，后端会复用同一条请求记录与 work item，不会把相同 demand 反复写成多条队列请求。
- 回退策略：
  - 未配置 AI（缺少 `AI_API_KEY`）：直接显示原文，不显示“翻译中”假状态。
  - AI 调用失败：显示原文，并允许用户重试。
- 请求复用：若某条 release 已经有 pending/ready/error 结果，后端会直接复用同一条结果记录；只有缺少当前 source hash 结果，或 pending 状态找不到活跃 work item 时，才会补建队列任务。
- 重试约束：自动可见窗口轮询不会把 `error` 条目重新入队；只有用户手动点击“翻译”这类显式重试，才会带上重试意图并把最近一次失败 request/work item 重置回队列，旧的更早历史 request 仍保留终态快照。
- 输出要求：保留版本号/仓库名/代码片段/专有名词，不做过度“意译”。

## 数据来源与同步

- 登录方式：GitHub OAuth。
- OAuth 职责：用于登录、读取和同步（Feed / Notifications / Starred / Releases / Followers / owned repo stargazers）。
- OAuth scope：采用最小授权策略（不包含用于站内反馈写操作的额外 scope）。
- Release 反馈写操作：使用用户提供的 GitHub PAT（Personal Access Token），与 OAuth 通道分离。
  - Fine-grained PAT：按 GitHub Reactions 接口可不额外申请 repository permissions，但 token 必须覆盖目标仓库。
  - Classic PAT：公共仓库建议 `public_repo`，私有仓库需 `repo`。
- 本地存储：SQLite（默认 `./.data/octo-rill.db`）。
- Release 数据语义：按“共享事实”处理，release 以稳定 `release_id` 引用；用户 Star 仅用于决定个人列表与同步范围。
- 同步数据：
  - Starred repos（用于确定关注范围）
  - Releases（共享事实数据；用于信息流、日报与详情）
  - Notifications（用于 Inbox 列表与侧栏快捷入口）
  - Followers（当前登录账号被谁关注，用于社交动态）
  - Owned repo stargazers（当前登录账号名下个人仓库被谁加星，用于社交动态）
- 日报回填：
  - 服务启动时会为每个用户先同步一次 releases，再补齐最近 7 天日报（已存在日期会跳过）
  - 启动时还会扫描命中过时格式签名的已标准化日报，并基于原窗口原位刷新正文与 memberships
- 可见性规则：
  - 取消 Star 后，该仓库可从当前用户的列表中消失；
  - 但历史日报中的 release 详情链接（`/?tab=briefs&release=<release_id>`）仍应可访问。
- 语义边界：
  - 列表与日报候选按 Star 过滤（个性化可见范围）。
  - 详情读取与详情翻译按 `release_id` 读取（共享事实读取），不依赖当前 Star 状态。

## Release 反馈授权策略

- 默认可用：所有 Release 卡片都展示反馈计数。
- 站内点按前置条件：当前用户已配置 PAT，且 PAT 对目标仓库可访问。
- 未配置或无权限时：反馈保持只读并显示引导信息；不影响 Feed 阅读与同步流程。
- 交互约束（当前任务补充）：
  - 点击反馈按钮且未配置 PAT：必须弹出 PAT 配置对话框（不能只在卡片底部报错）。
  - 对话框必须给出配置路径与最小权限说明（classic PAT：公共仓库 `public_repo`、私有仓库 `repo`）。
  - 输入 PAT 后自动可用性检查，防抖窗口为 `800ms`，且仅最后一次输入结果生效。
  - 检查状态必须可见（`idle/checking/valid/invalid`）；仅 `valid` 状态允许保存并继续。

## 非目标（本项目不做什么）

- 不做完整 GitHub 客户端：不在站内处理评论、合并、标记已读等操作。
- 不做多语言设置页：当前固定中文。
- 不做持续后台全量自动同步：仍以用户手动触发 Sync 为主（仅在服务启动回填日报时补做一次 releases 同步）。
